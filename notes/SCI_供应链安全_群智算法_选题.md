## 正向选题 1：群智多目标优化驱动的软件供应链漏洞传播链预测与优先级排序

**英文标题**：Swarm Multi-Objective Optimization for Vulnerability Propagation Chain Prediction and Prioritization in Software Supply Chains

---

### 1️⃣ 研究背景与动机

现代软件严重依赖开源组件——一个典型项目的依赖树深度可达 10+ 层，包含数百个直接和传递性依赖。当一个底层组件（如 Log4j）被发现漏洞时，**漏洞会沿依赖链向上传播**，影响所有直接或间接依赖它的项目。2025年Q1仅PyPI就发现了1.7万个恶意包，漏洞传播的规模和速度前所未有。

现有方案的瓶颈：
- **SBOM + 静态依赖分析**：只能告诉你"存在漏洞"，无法评估"漏洞在你的特定调用路径上是否真的可达（reachable）"
- **CVSS评分排序**：不考虑依赖链上下文，一个CVSS 9.8的漏洞如果在你的代码中从未调用相关函数，实际风险可能为0
- **人工审计**：面对数百个依赖的漏洞告警，安全团队疲于奔命（alert fatigue）

**核心问题**：如何在巨大的依赖图上，同时考虑漏洞可达性、利用难度、业务影响、修复成本等多个冲突目标，自动排序出真正需要优先修复的漏洞传播链？——这是一个典型的**大规模图上的多目标组合优化问题**，群智算法天然适合。

### 2️⃣ 核心创新点

1. **构建"漏洞传播图"（Vulnerability Propagation Graph, VPG）**：将SBOM依赖树与代码调用图融合，构建带权有向图——节点是组件/函数，边权重编码漏洞可达性概率（通过静态分析+LLM辅助判断函数调用是否触发漏洞代码路径）。

2. **设计多目标群智传播链搜索算法（MO-SwarmChain）**：基于改进的多目标蚁群优化（MOACO），在VPG上搜索"高危传播链"——每只蚂蚁在图上游走形成一条传播链，同时优化四个目标：漏洞可达性、利用难度（EPSS评分）、业务影响（组件关键度）、修复成本（版本升级兼容性）。输出帕累托最优的传播链排序。

3. **GNN增强的漏洞可达性预测**：使用图注意力网络（GAT）在依赖图上学习组件间的语义关系，预测漏洞在传递性依赖中的实际可达概率，作为蚁群搜索的启发式信息。

4. **增量式群智更新机制**：当新漏洞披露（CVE发布）时，无需重新搜索整个图，只需在受影响的局部子图上增量更新信息素，实现实时漏洞优先级动态调整。

### 3️⃣ AI 技术栈

- **图注意力网络（GAT）**：学习依赖图上的组件表征，预测漏洞可达性
- **大语言模型（LLM）**：辅助判断代码调用路径是否触发漏洞（理解API语义）
- **静态程序分析**：构建函数级调用图，提供可达性的精确分析

### 4️⃣ 群智算法的结构性角色

**使用算法**：改进的多目标蚁群优化（MOACO），引入自适应信息素策略和帕累托档案维护。

**精确位置**：MOACO是传播链搜索与排序的**核心引擎**——在VPG上搜索多目标最优的漏洞传播路径。

**不可替代性论证**：
- **vs 贪心排序（按CVSS）**：CVSS不考虑依赖链上下文，MOACO通过图上搜索发现真实的传播路径
- **vs 穷举搜索**：依赖图规模巨大（数千节点、数万边），穷举所有传播链不可行，MOACO通过信息素引导高效搜索
- **vs 图神经网络直接预测**：GNN可以预测单个节点的风险分数，但无法生成完整的"传播链"（路径级输出），MOACO在图上游走天然产生路径

### 5️⃣ 实验设计

| 项目 | 详情 |
|------|------|
| **数据集** | ①GitHub Advisory Database（真实CVE+依赖关系）②deps.dev（Google开源依赖图数据库）③Libraries.io（包依赖元数据）④选取Log4j/XZ Utils等真实供应链事件做案例研究 |
| **对比基线** | ①CVSS排序 ②EPSS排序 ③Snyk/Dependabot优先级 ④纯GNN节点风险评分 ⑤随机搜索 ⑥贪心路径搜索 |
| **评估指标** | 主要：Top-K精准率（vs 人工专家排序）、漏洞修复效率（修复前K个传播链后的风险降低率）；辅助：搜索时间、帕累托前沿覆盖度 |

## 正向选题 2：群智调度的LLM多智能体恶意开源包检测——对抗"Slopsquatting"新型攻击

**英文标题**：Swarm-Scheduled LLM Multi-Agent System for Malicious Open-Source Package Detection Against Slopsquatting Attacks

---

### 1️⃣ 研究背景与动机

2026年出现了一种全新的供应链攻击向量——**"Slopsquatting"（LLM幻觉包名攻击）**：大语言模型在生成代码时会"幻觉"出不存在的包名（如`pip install flask-caching-redis`），攻击者监控LLM高频幻觉的包名，抢先在PyPI/NPM上注册同名恶意包，当开发者复制LLM生成的代码并安装时就会中招。这种攻击与传统typosquatting不同——它利用的是**AI系统本身的缺陷**来投毒供应链。

2025年Q1仅PyPI就发现1.7万+恶意包，且攻击手法日益隐蔽（混淆代码、延迟触发、环境感知）。现有检测方案的局限：
- **规则引擎（Bandit/Safety）**：只能检测已知模式，无法识别新型混淆手法
- **单一ML模型**：对跨语言（PyPI vs NPM vs RubyGems）、跨攻击类型的泛化能力差
- **人工审计**：面对每天数百个新包上传，完全不可扩展

### 2️⃣ 核心创新点

1. **群智调度的多LLM智能体检测框架（SwarmGuard）**：部署N个专业化LLM智能体——包名分析智能体（检测typosquatting/slopsquatting）、代码语义智能体（理解包功能是否与描述一致）、行为分析智能体（分析install hook/post-install脚本）、元数据智能体（检查作者信誉/发布时间/下载量异常），通过改进的人工蜂群（ABC）算法动态调度各智能体的分析优先级和资源分配。

2. **基于群智信息素的恶意包传播追踪**：在包依赖图上维护"恶意信息素"——当一个包被判定为恶意时，信息素沿依赖链反向传播到所有依赖它的项目，自动标记受影响范围。信息素浓度随时间衰减（包被修复后风险降低），实现动态风险评估。

3. **LLM幻觉包名预测模型**：训练一个专门的模型预测"哪些不存在的包名最可能被LLM幻觉生成"，提前在包注册表中标记这些高风险名称空间，实现**预防性防御**而非被动检测。

4. **跨生态系统的群智协同检测**：PyPI/NPM/RubyGems/Maven等多个包注册表的检测智能体形成跨生态系统群智体，共享恶意模式知识（信息素跨生态迁移），使得在一个生态中发现的攻击模式可以立即用于保护其他生态。

### 3️⃣ AI 技术栈

- **大语言模型**：代码语义理解、包功能描述一致性判断
- **CodeBERT/UniXcoder**：代码片段向量化表征
- **异常检测**：Isolation Forest/DBSCAN检测元数据异常
- **序列模型**：分析install脚本的行为序列

### 4️⃣ 群智算法的结构性角色

**使用算法**：改进的人工蜂群（ABC）用于智能体调度 + 蚁群信息素用于恶意传播追踪。

**不可替代性论证**：
- **vs 固定流水线检测**：不同恶意包的攻击手法差异巨大，固定流水线无法自适应分配检测资源。ABC的三角色机制（侦察蜂探索新包、采蜜蜂深入分析可疑包、观察蜂全局决策）天然适合这种"大量包中找少量恶意"的场景
- **vs 纯LLM直接检测**：单次LLM调用的token成本高，面对每天数百个新包不可承受。群智调度通过优先级排序将LLM资源集中在最可疑的包上，降低70%+的API成本

### 5️⃣ 实验设计

| 项目 | 详情 |

| **数据集** | ①Backstabber's Knife Collection（恶意PyPI/NPM包数据集）②Datadog恶意包数据集 ③2025开源供应链投毒报告中的5.7万恶意包样本 ④自建LLM幻觉包名数据集 |
| **对比基线** | ①Bandit规则引擎 ②Cerebro (ACM 2025) ③跨语言恶意包检测 (Elsevier 2026) ④单LLM智能体检测 ⑤OSSGadget/Socket.dev ⑥随机调度多智能体 |
| **评估指标** | 主要：检测率、误报率、检测延迟（从包上传到检测的时间）；辅助：Token消耗效率、跨生态迁移准确率 |

## 反向选题 3：开源生态系统的"群落免疫"——用人工免疫群智机制实现供应链的自愈防御

**英文标题**：EcoImmune: Artificial Immune Swarm Intelligence for Self-Healing Software Supply Chain Ecosystems

---

### 💡 核心灵感：生物免疫系统 ≡ 软件供应链防御系统

这个选题的灵感来自一个深层的**生物学-软件工程同构关系**：

| 生物免疫系统 | 软件供应链生态 |
|-------------|-------------|
| 生物体 | 软件项目 |
| 细胞（正常组织） | 正常的开源组件 |
| 病原体（病毒/细菌） | 恶意包/漏洞组件 |
| 抗原（病原体表面标记） | 恶意包的行为特征签名 |
| 抗体（识别并中和病原体） | 检测规则/安全策略 |
| 先天免疫（快速非特异性响应） | 静态规则检测（Bandit/Safety） |
| 适应性免疫（慢速特异性响应） | AI模型学习新攻击模式 |
| 免疫记忆（二次感染快速响应） | 历史恶意包知识库 |
| 自我-非我识别 | 正常包 vs 恶意包判别 |
| **群体免疫（herd immunity）** | **整个生态系统的集体防御能力** |
| 免疫细胞的克隆选择与增殖 | 有效检测规则的复制与传播 |
| 细胞因子信号传导 | 安全告警在生态系统中的传播 |

**关键洞察**：现有供应链安全方案都是**"个体免疫"**——每个项目独立做安全检测。但生物界真正强大的防御是**"群体免疫"**——当足够多的个体具有免疫力时，即使少数个体未免疫也能被保护。

**本选题提出**：将整个开源生态系统（PyPI/NPM的数百万个包）视为一个"超级有机体"，用人工免疫系统（AIS）的群智机制实现**生态级的群体免疫**。

### 核心创新点

1. **供应链免疫细胞模型**：将每个开源项目的安全检测能力抽象为"免疫细胞"——
   - **树突细胞（Dendritic Cell, DC）**：监控包注册表的新包上传，执行初步筛选（先天免疫）
   - **T细胞**：对可疑包进行深度分析（适应性免疫），使用深度学习模型进行精确判定
   - **B细胞**：产生"抗体"（检测规则），并通过克隆选择机制让有效的抗体增殖传播到整个生态
   - **记忆细胞**：存储历史恶意包的特征，实现对变种的快速二次响应

2. **群智驱动的克隆选择与亲和力成熟**：使用改进的克隆选择算法（CSA，一种免疫群智算法）驱动"抗体"（检测规则）的演化——高亲和力的抗体（能准确检测恶意包的规则）被克隆和超变异，产生更多变体以覆盖恶意包的变种。**这个过程是完全自动化的——生态系统自动"进化"出针对新攻击的防御能力。**

3. **细胞因子信号的供应链传播模型**：当一个包被判定为恶意时，"细胞因子"（安全告警信号）沿依赖图传播——下游项目收到信号后自动提升免疫警戒级别（加强检测灵敏度），上游项目收到信号后自动检查是否被污染。信号强度随传播距离衰减（类似真实细胞因子的扩散方程），避免过度告警。

4. **群体免疫阈值理论**：借鉴流行病学的群体免疫阈值公式（HIT = 1 - 1/R₀），建立软件供应链的"群体免疫阈值"理论——当生态系统中具有有效检测能力的项目比例超过阈值时，恶意包的传播将被自然遏制。这为**供应链安全投资决策**提供了理论指导（需要保护多少比例的项目才能实现生态级安全）。

### 群智算法的角色——"免疫系统本身"

这里群智算法不是在优化某个安全模型，而是**构成了供应链免疫系统的运行逻辑**：
- 克隆选择算法 = 抗体（检测规则）的自动演化机制
- 免疫网络理论 = 检测器之间的协同与抑制关系
- 危险理论 = 区分正常更新与恶意行为的上下文感知

**没有群智免疫机制，整个防御系统就不存在。** 这是从"工具"到"架构"的质变。

### 不可替代性

- **vs 集中式安全扫描（Snyk/Dependabot）**：集中式方案是"中心化免疫"（依赖单一检测器），一旦检测器失效整个生态暴露。群智免疫是分布式的，个体失效不影响群体
- **vs 传统AI检测模型**：静态模型无法自动适应新攻击模式，群智免疫通过克隆选择实现"免疫记忆+自动进化"，天然适应未知威胁
- **vs 区块链信任方案**：区块链解决的是"信任传递"问题，不解决"恶意检测"问题。免疫群智同时解决检测和传播防御

### 实验设计

| 项目 | 详情 |
|------|------|
| **数据集** | ①Libraries.io完整依赖图（3200万+包）②PyPI/NPM恶意包历史数据 ③模拟供应链攻击传播场景（基于真实依赖图注入恶意节点）|
| **对比基线** | ①Snyk集中式扫描 ②Socket.dev ③纯深度学习检测（无免疫机制）④随机防御部署 ⑤基于PageRank的关键节点保护 |
| **评估指标** | 主要：恶意包传播遏制率、群体免疫达成时间、新攻击自适应速度；辅助：免疫细胞多样性指数、克隆选择收敛速度、误报率 |

## 反向选题 4：依赖生态的"共生与寄生"——用群智共生演化模型预测供应链的系统性风险涌现

**英文标题**：SymbioChain: Swarm Symbiotic-Parasitic Co-Evolution for Predicting Systemic Risk Emergence in Software Supply Chain Ecosystems

---

### 💡 核心灵感：生态学中的共生-寄生动态 ≡ 开源依赖生态的健康-恶意动态

**这是所有选题中最"偏门"的一个。**

生态学中，物种之间存在三种关系：
- **互利共生（Mutualism）**：双方受益（如蜜蜂与花）
- **寄生（Parasitism）**：一方受益、另一方受害（如寄生虫与宿主）
- **竞争（Competition）**：双方争夺资源

**映射到开源供应链**：
- **互利共生** = 正常的开源依赖关系（项目A使用库B的功能，库B获得用户和贡献者）
- **寄生** = 恶意包/投毒（恶意包伪装成正常包，窃取宿主项目的数据/资源）
- **竞争** = 功能相似的包之间的竞争（如requests vs urllib3）

**关键洞察**：现有供应链安全研究只关注**个体层面**（单个恶意包的检测），完全忽视了**生态层面的系统性风险**——当整个依赖生态的"寄生率"超过某个临界点时，生态系统会发生**相变（phase transition）**——从健康状态突然崩溃（类似生态学中的"生态系统临界转变"）。

**本选题提出**：用群智共生演化算法模拟开源生态系统的"共生-寄生"动态，**预测系统性风险的涌现临界点**——在崩溃发生之前发出预警。

### 核心创新点

1. **开源生态的共生-寄生网络模型（Symbio-Parasitic Network, SPN）**：将依赖图中的每条依赖关系标注为"共生"（正常使用）或"潜在寄生"（存在风险），构建带有共生/寄生属性的加权网络。使用 GNN 学习节点表征，预测每条依赖关系从共生转变为寄生的概率。

2. **群智共生演化模拟器（Symbiotic Swarm Simulator, S³）**：借鉴生态学的 Lotka-Volterra 方程（捕食者-猎物动态），使用群智算法模拟开源生态的演化——正常包种群（宿主）和恶意包种群（寄生者）在依赖网络上共演化。群智算法的每个个体代表一种可能的生态演化路径，种群通过选择压力收敛到最可能的未来生态状态。

3. **系统性风险的临界点预测**：通过群智模拟发现生态系统的**"临界寄生率"**——当恶意包占比超过此阈值时，信任链将发生级联崩溃（类似银行系统的系统性风险）。这个临界点取决于依赖图的拓扑结构（如是否存在"超级节点"——被数万项目依赖的核心库），为**生态系统级的安全资源分配**提供理论依据。

4. **"关键物种"识别与保护策略**：借鉴生态学的"关键物种"（keystone species）概念——某些开源包虽然自身不大，但其被攻陷会导致整个生态级联崩溃（如left-pad事件、XZ Utils事件）。使用群智搜索在依赖网络上识别这些"关键物种"，并设计最优的保护资源分配策略（有限安全审计预算下，优先保护哪些包？）。

5. **生态多样性与安全韧性的关系**：借鉴生态学的"多样性-稳定性"假说，研究开源生态的"组件多样性"（同一功能是否有多个替代包）与供应链安全韧性的关系。群智算法模拟在不同多样性水平下生态系统对攻击的抵抗能力，为"避免过度依赖单一组件"提供量化建议。

### 群智算法的角色——"生态演化的模拟引擎"

群智算法在这里扮演的是**生态系统演化的模拟器**——它不是在优化某个目标函数，而是在模拟一个复杂自适应系统（Complex Adaptive System, CAS）的涌现行为。每个群智个体代表一种可能的生态演化路径，种群的统计特性揭示了系统性风险的概率分布。

### 不可替代性

- **vs 静态依赖图分析**：静态分析只能看到当前快照，无法预测未来演化趋势。群智模拟可以推演"如果核心库被攻陷，生态会如何演变"
- **vs 蒙特卡洛模拟**：蒙特卡洛是无引导的随机采样，在高维演化空间中效率低下。群智算法通过种群协同和选择压力，高效采样最可能的演化路径
- **vs 传统流行病学模型（SIR/SIS）**：SIR模型假设均匀混合（每个节点等概率感染），不适用于高度异质的依赖网络。群智演化直接在真实网络拓扑上运行，自然捕获网络异质性

## 反向选题 5：供应链的"蚂蚁侦察兵"——用群智觅食理论实现依赖更新的最优安全迁移路径搜索

**英文标题**：AntMigrate: Ant Foraging Theory-Inspired Optimal Security Migration Path Search for Software Dependency Updates

---

### 💡 核心灵感：蚂蚁觅食的路径优化 ≡ 依赖版本升级的安全迁移路径搜索

**这个选题切入了一个被严重忽视的实际痛点**：当一个依赖库被发现有漏洞时，升级到安全版本并不是简单的`npm update`——版本升级可能引入不兼容的API变更、破坏现有功能、甚至引入新的漏洞（从一个坑跳到另一个坑）。在复杂的依赖树中，升级一个底层库可能触发"依赖地狱"（dependency hell）——数十个中间依赖需要同步升级，每个升级决策都有兼容性风险。

**蚂蚁觅食的同构映射**：
- **蚁巢** = 当前的漏洞版本配置（起点）
- **食物源** = 安全的目标版本配置（终点）
- **路径** = 一系列版本升级步骤（中间状态）
- **路径长度** = 迁移成本（兼容性破坏风险 + 工作量）
- **障碍物** = 不兼容的版本组合（死路）
- **信息素** = 历史成功迁移经验（社区中其他项目的升级记录）
- **最短路径的涌现** = 最优安全迁移策略的自动发现

### 核心创新点

1. **依赖版本空间的图建模**：将所有可能的依赖版本组合建模为一个巨大的"版本空间图"——每个节点是一个完整的依赖配置（所有包的版本号组合），边代表单步版本变更。在这个图上搜索从"当前漏洞配置"到"安全配置"的最优路径。

2. **蚁群觅食驱动的迁移路径搜索**：使用改进的蚁群优化（ACO）在版本空间图上搜索最优迁移路径——每只蚂蚁从当前配置出发，按信息素和启发式信息（版本兼容性评分、安全评分）选择下一步升级，最终到达安全配置。多只蚂蚁的路径汇聚后，涌现出兼容性风险最小、安全增益最大的迁移策略。

3. **LLM赋能的兼容性预测**：使用大语言模型分析changelog/release notes/API diff，预测版本升级的兼容性风险（"从v2.3升级到v3.0会破坏哪些API？"），作为蚁群搜索的启发式信息。

4. **社区迁移经验的信息素建模**：从GitHub上百万个项目的依赖更新历史中提取"社区迁移信息素"——如果很多项目成功从v1.x升级到v2.x，这条路径的信息素浓度就高（经过社区验证的安全路径）。这利用了群智的"集体智慧"——百万开发者的升级决策构成了一个巨大的"蚁群"。

5. **多目标迁移优化**：同时优化安全性（消除所有已知漏洞）、兼容性（最小化API破坏）、最小变更（尽量少改动依赖版本），输出帕累托最优迁移方案集合。

### 群智算法的角色——"集体迁移智慧的数学化"

这个选题最精妙之处在于：**开源社区本身就是一个"蚁群"**——数百万开发者的依赖升级决策就是蚂蚁的觅食路径。本选题不是"人为设计"一个蚁群来搜索，而是**将真实社区行为数学化为蚁群模型**，然后用ACO算法加速和优化这个自然过程。

### 不可替代性

- **vs Dependabot自动升级**：Dependabot只做单步升级（一次升一个包），不考虑多包联动的兼容性。ACO搜索整条迁移路径，考虑全局兼容性
- **vs 约束求解器（SAT/SMT）**：版本空间是指数级的，精确求解不可行。ACO通过启发式搜索在多项式时间内找到近似最优解
- **vs 人工迁移规划**：复杂项目的依赖树有数百个节点，人工规划不可扩展。ACO自动化整个过程

### 实验设计

| 项目 | 详情 |
|------|------|
| **数据集** | ①GitHub依赖更新历史（dependabot PR记录）②npm/PyPI版本兼容性数据 ③真实迁移案例（Log4j 1.x→2.x、Python 2→3相关依赖迁移）|
| **对比基线** | ①Dependabot逐步升级 ②Renovate Bot ③贪心升级（每次选最新版本）④随机搜索 ⑤SAT求解器（小规模精确解）|
| **评估指标** | 主要：迁移成功率（升级后测试通过率）、漏洞消除率、API破坏数量；辅助：搜索时间、路径长度、社区信息素利用率 |
